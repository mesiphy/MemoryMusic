## 1. 计划目标

MemoryMusic 当前目标不是建设完整音乐播放器，而是尽快验证以下核心闭环：

> 收录一首歌曲 → 低成本记录个人线索 → 用任意记忆点找回歌曲 → 调用网易云音乐播放。

本行动计划以个人自用 MVP 为范围，优先交付可验证的纵向功能，不提前建设社区、推荐算法、多设备同步或完整流媒体播放能力。

### 1.1 MVP 达成标准

当以下条件同时满足时，视为 MVP 核心能力成立：

- 能手工添加或从网易云导入至少 100 首歌曲；
- 能为歌曲添加多个标签、多条感悟、事件和自定义别名；
- 能统一搜索歌名、歌手、专辑及全部个人字段；
- 搜索结果能说明命中了哪些字段；
- 能从搜索结果唤起网易云音乐或可靠地打开歌曲页面；
- 能备份并恢复个人数据库，且个人数据不会因平台歌曲失效而丢失；
- 20 个真实“忘歌任务”中，至少 80% 能在 10 秒内找回目标；
- 主分支持续通过 Lint、格式检查、类型检查、测试和生产构建。

## 3. 开发原则

### 3.1 优先完成纵向闭环

每个阶段都应交付可操作、可验证的用户路径。数据表、接口或页面单独存在不算完成，必须至少能被下一层调用并通过自动化或人工验收。

### 3.2 本地优先

- 标签、感悟、事件等个人数据默认只保存在本地；
- 数据库应支持备份、恢复和开放格式导出；
- 网易云数据只作为外部内容引用，不拥有个人数据的生命周期；
- 平台歌曲不可见或被删除时，只更新状态，不级联删除个人记录。

### 3.3 适配器隔离

- 数据同步通过 `NeteaseDataAdapter` 等接口与领域层隔离；
- 播放通过 `PlaybackAdapter` 与搜索、标注逻辑隔离；
- Cloud 测试使用 Mock 适配器；
- Windows 专属能力必须在本地真机验收。

### 3.4 不碰版权与账号红线

- 不破解付费、版权、地域或 DRM 限制；
- 不把第三方非公开接口作为唯一正式依赖；
- 不在 Git、日志、数据库明文或前端代码中保存私钥；
- 优先评估网易云官方开放平台及官方工具。

### 3.5 每次只解决一个主要风险

开发顺序依次验证：

1. 数据能否可靠保存；
2. 个人线索是否容易录入；
3. 搜索是否真的提高找回率；
4. 播放跳转是否稳定；
5. 网易云同步是否值得维护。

## 7. Cloud 与本地开发分工

| 工作类型 | Codex Cloud | Windows 本地 |
| --- | --- | --- |
| TypeScript 领域模型 | 主要开发与测试 | 集成复核 |
| SQLite 迁移与 Repository | 主要开发与测试 | 文件与性能验证 |
| Vue 页面与普通交互 | 可开发和构建 | 最终体验验证 |
| FTS5 搜索与测试集 | 主要开发与测试 | 中文输入体验验证 |
| 文档、重构、静态检查 | 适合 | 复核 |
| `orpheus://` | 仅实现接口和 Mock | 必须真机验证 |
| Windows SMTC | 仅实现抽象和 Mock | 必须真机验证 |
| 网易云桌面客户端 | 不验证 | 必须真机验证 |
| Credential Manager | 不验证 | 必须真机验证 |
| Windows 安装包 | 可做配置检查 | 必须本地安装验证 |

### 7.1 本地交给 Cloud

```text
本地功能分支 → 完成检查 → commit → push → 从该远端分支启动新 Cloud 任务
```

任务提示应包含：

- 分支名和最新提交号；
- 本轮唯一主要目标；
- 允许和禁止修改的范围；
- 必须运行的验证命令；
- Windows 专属能力只能使用 Mock 的限制。

### 7.2 Cloud 结果接回本地

```text
查看 Cloud diff → 应用到干净功能分支 → 本地检查 → Windows 验证 → commit → push
```

不要把 Cloud 任务显示为 `READY` 等同于代码已经进入仓库；只有本地审查、验证、提交并推送后，才算进入产品基线。

## 8. 每个任务的完成定义

任何开发任务只有满足以下条件才能标记完成：

- 功能范围与任务描述一致，没有夹带无关重构；
- 新行为有自动化测试，或明确记录无法自动化的原因；
- 数据迁移和删除行为经过测试；
- 错误场景有用户可理解的反馈；
- 不在日志、截图、测试夹具和提交中泄露凭据；
- 通过以下检查：

```text
pnpm lint
pnpm format:check
pnpm typecheck
pnpm test
pnpm build
```

- Windows 专属功能完成真机验证；
- 相关 README、`AGENTS.md` 或设计说明已经更新；
- Git 工作树干净，提交信息能解释本次改动；
- 已推送远端，重要功能有可回退提交点。

## 9. 风险控制

| 风险 | 早期信号 | 行动 |
| --- | --- | --- |
| 标注成本过高 | 记录一次超过 10 秒、连续数日不记录 | 优先做快捷键、最近标签和一句话保存 |
| 搜索价值不足 | 真实忘歌测试仍需翻列表 | 补充缺失字段与命中解释，不先堆 AI |
| 数据模型过度设计 | 表很多但没有可操作界面 | 每个模型必须服务当前纵向闭环 |
| 网易云接口不稳定 | 同步频繁失败或字段变化 | 适配器隔离、离线可用、网页兜底 |
| Windows 集成误判 | Cloud 测试通过但真机失败 | Windows 功能单列验收清单和版本记录 |
| Native 依赖安装失败 | Cloud 或新机器无法 `pnpm install` | 锁定版本、限制 `allowBuilds`、验证干净安装 |
| 个人数据丢失 | 升级后记录缺失、恢复不可用 | 先备份再迁移，建立恢复测试 |
| 范围持续膨胀 | 同时开发推荐、社交、移动端 | 用 P0/P1/P2 队列和 MVP 指标约束 |

## 11. 下一次决策检查点

完成 M2 后进行第一次产品决策检查：

- 如果个人线索明显提高找回率，继续投入播放适配和收藏同步；
- 如果搜索有效但记录成本过高，优先投入快速记录，而不是增加搜索算法；
- 如果记录成本可接受但找回率低，重新检查数据字段与搜索解释；
- 如果个人自用频率仍很低，暂停平台同步和复杂功能，重新评估痛点强度。

行动计划每完成一个里程碑更新一次，所有新增功能必须说明它提高了找回率、降低了记录成本、保护了个人数据，或降低了外部依赖风险中的哪一项。
