# 数据安全与导出

本文记录 MemoryMusic M5 的数据库备份、恢复和 JSON 导出边界。目标是让个人资料既能恢复到全新的数据库文件，也能脱离应用独立阅读。

## 1. 数据版本

- 当前 SQLite schema 版本由 `PRAGMA user_version` 与 `schema_migrations` 共同记录，两者必须连续且一致；
- 当前 JSON 导出格式名为 `memory-music-export`，格式版本为 `1`；
- 数据库备份可以来自当前版本或更早的受支持版本；恢复暂存阶段会先升级副本，不修改用户选择的原备份；
- 高于当前 schema 版本的备份会被拒绝，并提示先升级应用。

数据库 schema 版本与 JSON 格式版本相互独立。以后数据库迁移不必自动提高导出格式版本；只有 JSON 的字段语义或兼容性发生变化时才提高导出格式版本。

## 2. 备份

### 手动备份

“备份与导出”面板中的“立即备份”会打开系统保存对话框。主进程使用 SQLite 在线备份 API 生成一致快照，并在替换目标文件前验证：

- `integrity_check` 为 `ok`；
- `foreign_key_check` 没有失败项；
- MemoryMusic 核心表存在；
- `schema_migrations` 连续且与 `user_version` 一致。

渲染进程不能传入任意路径或直接读取数据库，只能调用类型化的备份命令。

### 自动备份

- 应用每天按 UTC 日期最多创建一份自动备份；
- 自动备份保存在 Electron `userData/backups` 目录；
- 只轮换严格匹配 `memory-music-auto-*.sqlite3` 的文件，不会删除目录中的手动文件；
- 默认保留最新 7 份；
- 已有数据库在迁移前先备份；新数据库完成初始化后再建立第一份自动备份；
- 恢复开始前另建一份 `memory-music-pre-restore-*.sqlite3`，不占用 7 份日常轮换额度。

## 3. 恢复

恢复采用“验证、暂存、重启应用”的流程，避免覆盖正在被 Electron 主进程和 WAL 使用的文件：

1. 用户选择 `.sqlite3` 或 `.db` 文件；
2. 主进程只读检查完整性、关系、核心表和版本；
3. 系统确认对话框明确提示将替换资料并重启；
4. 当前数据库先生成恢复前备份；
5. 所选备份复制到内部临时文件并迁移到当前 schema；
6. 迁移结果再次通过完整性检查后写入 `userData/recovery/restore-pending.sqlite3`；
7. IPC 成功返回后应用重启；
8. 下一次打开数据库前，主进程移走旧数据库及其 WAL/SHM 文件，再安装已验证的暂存文件。

任何选择、验证或暂存失败都不会替换当前资料。启动时如果暂存文件意外损坏，会将其隔离为 `restore-rejected-*.sqlite3`，继续使用原数据库并显示警告。

## 4. JSON 导出格式 v1

导出文件为 UTF-8、两空格缩进并以换行结束。顶层结构如下：

```json
{
  "format": "memory-music-export",
  "formatVersion": 1,
  "schemaVersion": 4,
  "exportedAt": "2026-07-19T08:00:00.000Z",
  "data": {}
}
```

`data` 包含：

- `tracks` 与 `providerTracks`；
- `tags` 与 `trackTags`；
- `notes`；
- `memories` 与 `memoryTracks`；
- `cues`，包括别名、错记歌词和其他召回线索；
- `quickCaptureInbox`；
- `syncStates`；
- `searchQueryLog`。

关系使用显式 ID 字段表示，数组按稳定键排序。`search_documents` 和 FTS 表属于可重建派生数据，不进入导出，也不会在导出过程中修改任何原始资料。

## 5. 自动化恢复证据

M5 回归测试覆盖：

- 活跃 WAL 数据库在线备份后，歌曲、平台映射、标签关系、感悟、事件关系、召回线索、待整理箱、同步状态和无结果查询日志完整恢复到全新数据库；
- 恢复后的新文件通过 `integrity_check`、`foreign_key_check`、关闭重开和当前版本迁移；
- v3 数据在应用迁移前先形成备份，备份副本可升级并保留个人记录；
- 无效文件和更高版本文件被拒绝，当前资料与恢复暂存均不改变；
- 损坏的启动暂存被隔离，当前数据库保持可用；
- 9 个日备份轮换后仅保留最新 7 个；
- JSON 可由标准解析器直接读取，包含原始资料与关系且排除派生搜索索引；
- Vue 交互只在确认完成后报告成功，失败和取消不会误报。

## 6. Windows 目录成品验收（2026-07-20）

在 `release/win-unpacked/MemoryMusic.exe` 中使用独立 `--user-data-dir` 完成：

1. 新建“恢复基线歌曲”，系统保存对话框生成 `baseline.sqlite3`；
2. JSON 保存对话框生成 `export.json`，标准解析器读到格式 `memory-music-export` v1、schema v4 和 1 首基线歌曲，且不存在 `searchDocuments`；
3. 备份后新增“恢复后应消失”，使当前库变为 2 首；
4. 从 `baseline.sqlite3` 恢复，原生确认框明确提示替换、保护备份和重启；
5. 恢复前保护备份包含 2 首；重启后立即生成的日备份只包含 1 首基线歌曲；
6. 恢复后的日备份 `integrity_check=ok`、`foreign_key_check` 为 0 项、schema v4；
7. 重启进程继续携带同一隔离目录参数，没有读写正式资料库。

恢复完成后，同一隔离库又执行了 791 首收藏导入，因此后续当前库为 792 首；恢复完成瞬间生成的 1 首日备份与恢复前 2 首保护备份共同固定了恢复结果。验收还发现 SQLite 只读验证可能留下空的 pending WAL/SHM 文件，随后增加了清理逻辑和自动化断言。
