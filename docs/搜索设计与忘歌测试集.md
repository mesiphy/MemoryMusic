# MemoryMusic 搜索设计与忘歌测试集

> 版本：V0.1  
> 日期：2026-07-15  
> 对应里程碑：M2 统一个人搜索

## 1. 目标与边界

M2 搜索服务于“记不清歌名，但还记得个人线索”的场景。搜索只读取本地个人资料，不调用外部平台，不修改歌曲、标签、感悟、事件或召回线索原文，也不引入向量数据库或在线大模型。

## 2. 索引结构

SQLite v3 增加三个派生对象：

- `search_documents`：每首歌一行，保存客观字段、个人字段聚合值、规范化值和用于短词匹配的扁平文本；
- `search_documents_fts`：使用 SQLite FTS5 `trigram` tokenizer 的无内容全文索引；
- `search_query_log`：仅在没有结果时记录原始查询、规范化查询、匹配模式和用户补充的缺失字段。

索引字段映射如下：

| 搜索字段 | 数据来源 | 结果说明 |
| --- | --- | --- |
| 歌名 | `tracks.title` | 命中歌名 |
| 歌手 | `tracks.artist` | 命中歌手 |
| 专辑 | `tracks.album` | 命中专辑 |
| 别名 | `aliases.kind = alias/user` | 命中别名 |
| 错记歌词 | `aliases.kind = lyric` | 命中错记歌词 |
| 其他线索 | `aliases.kind = cue` | 命中其他线索 |
| 标签 | `tags` + `track_tags` | 命中标签 |
| 感悟 | `notes.body` | 命中感悟 |
| 事件 | 事件名称、描述、时间、地点和人物 | 命中事件 |

每次搜索前重建派生索引，优先保证 MVP 规模下的数据一致性。重建事务只清空并写入搜索表；自动化测试会在重建前后逐表比较全部原始个人数据。

## 3. 规范化与匹配策略

查询和索引字段统一执行：

1. Unicode NFKC 规范化，将全角拉丁字母、数字和兼容字符转为标准形式；
2. 使用中文区域规则转为小写；
3. 将换行、制表符、全角空格、不换行空格等常见空白合并为单个半角空格；
4. 去除首尾空白并按空格拆分查询词。

当所有查询词都不少于三个 Unicode 字符时使用 FTS5 trigram；任一查询词不足三个字符时使用转义后的参数化 `LIKE` 包含匹配。FTS 表达式对用户输入加引号，不把输入作为 SQLite 查询语法执行；异常表达式会安全降级到包含匹配。

## 4. 命中解释与排序

结果返回最多十条字段级命中原因，例如“命中标签：深夜”或“命中事件：青海湖自驾 · 日落后沿湖开车”。排序规则固定为：

1. 命中标签、感悟、事件、别名、错记歌词或其他个人线索；
2. 规范化后与歌名完全相同；
3. 命中原因数量更多；
4. 最近编辑时间更晚；
5. 歌曲 ID 更小，作为最终稳定排序键。

这使产品差异化的个人记忆优先于普通元数据，同时保证相同数据重复搜索得到相同顺序。

## 5. 简繁体与拼音评估

M2 暂不自动扩展简繁体、拼音全拼和拼音首字母，原因如下：

- NFKC 不负责简繁转换，自动转换需要额外词典和多音字策略；
- 拼音首字母容易产生大量短词冲突，可能降低个人字段优先排序的可解释性；
- 当前资料量目标约 100 首，先通过无结果日志判断真实需求，比预先扩大索引更可靠；
- 后续可在 `search_documents` 增加纯派生的 `traditional`、`pinyin_full` 和 `pinyin_initials` 列，不需要修改原始个人数据。

当无结果日志中简繁体或拼音查询达到可观察频率时，再以独立迁移和测试集加入；本阶段不使用向量搜索掩盖字段缺失。

## 7. 无结果闭环

没有结果时，应用自动在本地写入查询记录，并允许用户选择原本记得的是歌名、歌手、专辑、别名、歌词、标签、感悟、事件或其他线索。该反馈用于后续补资料和评估简繁体、拼音等扩展，不会自动生成或改写个人内容。
